目标能力

Agent A 在群里说“@AgentB …”后，系统能识别并投递给 Agent B。
Agent B 的响应能回到同一个群时间线，而不是只留在自己私有会话。
有回环保护，避免 A@B、B@A 无限互调。
Prompt 明确告诉 Agent：成员列表来自本系统，不要去查飞书成员接口。
实现方案（建议分两阶段）

阶段一：先打通可用闭环（建议 1 天）
在 chat 路由注入群成员目录上下文。
改 route.ts:250 附近 nativePrompt，新增 GroupMembers 段：agentId、显示名、可 @ 别名。
效果：Leader Rui 不会再说“去查飞书成员列表”，而是直接按已知成员发起 @。

后端增加 assistant 文本 @ 解析器。
在 route.ts:250 处理 responseText 后，解析 @xxx（支持 @id、@显示名）。映射目标必须在 currentGroup.members 内。

新增“转发而非直接执行”模式（MVP）。
检测到 @ 后，不立即递归调用模型，而是把一条系统转发消息写入目标 Agent 会话：
“来自 AgentA 的群内呼叫：…”。
写入可复用 session-runtime.ts:275 的 appendSessionMessage。
效果：先保证消息能送达，避免首版就遇到递归风暴。

阶段二：升级成自动协作（建议 1-2 天）
在后端实现受控递归调度。
当 A @ B 时，后端内部再调用一次 chat 流程给 B，附带：

relayDepth（初始 0，最大 2）
relayChain（已访问 agent 列表）
relayFrom（来源 agent）
relayGroupId / channelId
超过阈值或重复访问直接停止。
引入群时间线存储（关键）。
当前 UI 只拉一个 Agent 的消息，不够用。应新增“群时间线”存储（groupId + channelId 维度），所有用户消息、Agent回复、转发事件都写入该流。
读取时群聊页面只读这条群时间线，不再绑单个 loadAgentId。
现有读取入口在 page.tsx:169，会改成群维度读取。

UI 展示增加消息来源标签。
在群消息卡片显示 fromAgent、toAgent（若为转发），让协作链路可见。

安全与稳定策略（必须）
单次回复最多触发 2 个 @ 目标。
最大 relayDepth = 2。
同一条消息的 messageId 去重，防重复投递。
仅允许 @ 群成员，@外部名称直接忽略并提示。
加 3 类日志：mention_parse、relay_dispatch、relay_drop。
你提到的场景如何落地
用户说：@Leader Rui 你能向其余两位群成员发送消息吗
Leader 回复中若包含：@AgentB @AgentC 请同步状态
系统会：

解析出 AgentB、AgentC
写入群时间线一条“Leader 发起呼叫”事件
分别触发对 B/C 的受控调度
B/C 回复再写回同一群时间线
前端看到完整链路，不需要切会话
验收标准

群内 3 Agent，A @ B 后，B 在 5 秒内有回复并显示在同一群窗口。
A 一次 @ 两人，两个目标都能收到且各自回复。
构造 A@B、B@A，不会无限循环。
Leader 不再输出“无法获取飞书群成员列表”这类偏离行为。
